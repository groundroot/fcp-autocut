# FCP AutoCut v0.12.0 — 전면 구조 교체 플랜

작성: 2026-07-07 · 상태: 승인 대기

## 1. 왜 바꾸는가 (문제의 본질)

v0.11까지의 오류(오브젝트 삭제·자막 어긋남·갭·경계 이탈)는 개별 버그가 아니라
**구조적 원인**에서 나왔다: 앱이 타임라인의 **모든** 구성요소(커넥티드 클립, 보조
스토리라인, 중첩, 싱크 내부, 기존 타이틀…)를 이해하고 보존·리매핑하려 했다.
FCP 타임라인의 표면적은 사실상 무한하므로, 이 방향은 새 프로젝트 유형마다 새
엣지를 낳는다. 야간 검증으로 12종을 안정화했지만, **표면적 자체를 줄이지 않으면
반복**된다.

## 2. 새 원칙 (사용자 확정)

> **앱은 "오디오/영상을 분석해 자막화·무음컷하는 편의 도구"다.
> 분석 기준은 오직 메인 스토리라인(스파인 직속)의 오디오다.**

1. **분석 대상 = 메인 스토리라인의 오디오 소스만.** 클립마다 소스가 달라도 됨(다중 소스 OK).
2. **커넥티드 클립의 오디오는 분석에서 제외.** (컷 기준에도 영향 없음)
3. 타임라인 복잡성에 대응하려던 코드·기획은 **전면 철회** — 이해 못 하는 요소는
   "건드리지 않고 통과(pass-through)"가 기본 동작. **에러로 죽지 않는다.**
4. macOS·FCP **버전 전방위 대응** (입력 버전 보존 + 해당 DTD 검증 — 기존 자산 유지).
5. 결과: 심플하고 예측 가능한, "무조건 잘 작동"하는 앱.

## 3. 지원 매트릭스 (v0.12.0 계약 — 문서·코드·테스트의 단일 기준)

| 타임라인 요소 | 분석(오디오) | 자막 배치 | 무음컷 처리 |
|---|---|---|---|
| 스파인 직속 asset-clip | ✅ 기준 | ✅ | ✅ 침묵 구간 컷 |
| 스파인 직속 sync-clip | ✅ (주 소스) | ✅ | ✅ |
| 스파인 직속 gap / clip(b롤) / video | ❌ | ❌ | 위치만 이동 (통과) |
| 스파인 직속 mc-clip / ref-clip | ❌ | ❌ | 위치만 이동 (통과 + 안내 로그) |
| **커넥티드(lane) 전부** — 오디오 포함 | **❌ (원칙 2)** | ❌ | **부모 조각에 통짜 승계** (트리밍 없음) |
| 보조 스토리라인 (spine lane) | ❌ | ❌ | 부모 조각에 통짜 승계 |
| 기존 title/caption | — | 제거 후 재생성 (기존 정책) | 재생성 |

"통짜 승계" = 부모가 잘려도 자식은 자르지 않고, 첫 겹침 조각에 원위치로 1회 부착.
자식 내부의 미세 트리밍·start 전진·부분 겹침 계산 코드는 **전부 삭제**한다.
(잘린 b롤이 살짝 길게 남는 것은 허용 — 사용자가 다듬는 1초가, 앱이 깨지는 것보다 낫다)

## 4. 새 아키텍처 — 3계층 분리

```
[A. 스캔]   FCPXML → AudioTrackModel (읽기 전용, 원본 불변)
            스파인 직속 항목만: (kind, ref, tl_start, tl_end, src_file, src_start)
            오디오 소스 해석 실패 항목 → kind="opaque" (통과 대상)

[B. 분석]   AudioTrackModel → SpeechData
            항목별 오디오 추출 → VAD → ASR → 단어 타임스탬프 (Title Bible ①고정)
            ※ 이 계층은 FCPXML을 전혀 모름 — 순수 (파일, 구간) → (단어, 시간)

[C. 산출]   ① 자막본: 원본 트리 그대로 + 스파인 직속 클립에 title 삽입 (현행 유지)
            ② 무음컷본: 유지구간 테이블 [(tl_a, tl_b)] → 스파인 항목 슬라이스
               - 오디오 항목: 구간대로 분할 (start 재계산 — v0.11 검증 자산 유지)
               - opaque/gap/b롤: 위치만 리매핑 (분할하되 내용 미세조정 없음)
               - lane 자식: 통짜 승계 (§3)
```

핵심 감축: 현 `retranscribe.py`의 **커넥티드 수집·커넥티드 자막 배치·lane 트리밍·
sync-clip 내부 해석 특수 로직**이 사라지고, [A]의 단순 스캔과 [C]의 두 규칙
(오디오=분할 / 나머지=통과·승계)만 남는다.

### 유지하는 검증 자산 (v0.11에서 이미 견고)
프레임 타임베이스(`_frame_time_str`) · 경계 연속 승계(pass 0) · 미디어 범위 클램핑 ·
스냅 후 재클램프(자막 경계) · DTD 자동 검증 · Title Bible 분리 규칙 · UID 재부여 ·
언어별 폰트 · per-asset 오디오 추출.

## 5. 순차 실행 플랜

### Phase 1 — 스캔 계층 신설 (A)
- `silence_cutter/timeline_scan.py` 신설: `scan_main_storyline(root) → [Item]`
  - Item: `kind(audio|opaque)`, `element`, `tl_start`, `tl_end`, `media_path`,
    `src_start` (경계는 연속 승계로 정규화)
  - asset-clip/sync-clip → audio (소스 해석 실패 시 opaque로 강등 + 로그)
  - 그 외 전부 opaque. **어떤 태그도 에러를 내지 않는다.**
- 검증: 픽스처 12종 전부에서 스캔 결과 스냅샷 테스트 (항목 수·경계 연속성)

### Phase 2 — 분석 계층 분리 (B)
- 기존 VAD/ASR/자막분리 코드를 `Item` 리스트 소비 형태로 정리 (FCPXML 의존 제거)
- 커넥티드 관련 분석·수집 코드 삭제 (clip_items의 lane 수집, conn_tl 변환 등)
- 검증: Bible 7종 + 자막 타이밍 회귀 (기존 테스트 그대로 통과해야 함)

### Phase 3 — 자막본 산출 (C-①)
- 현행 결과물1 로직을 스캔 계층 위로 이식. 커넥티드 자막 배치 삭제
  (커넥티드 오디오를 분석 안 하므로 자막도 없음 — 원칙 2의 자연 귀결)
- 검증: 픽스처 12종 자막본 (컷·오브젝트 보존 + DTD + 자막 경계)

### Phase 4 — 무음컷 산출 단순화 (C-②)
- `_build_silence_removed` → `build_silence_cut(items, keep_table)` 재작성:
  - keep_table은 audio 항목의 VAD에서만 계산
  - audio: 분할(+start 재계산·클램핑) / opaque: 리매핑 분할(내용 무조정) /
    lane: 통짜 승계 일원화 (whole_attach 규칙을 유일 규칙으로)
- 삭제: lane 부분 트리밍, 자식 start 전진, sync-clip 특수 앵커 경로 등
- 검증: 픽스처 12종 무음컷본 (오브젝트 보존 + 연속성 + DTD) + 실제 커피국수

### Phase 5 — 실패 불가능화 (오류 → 통과 + 안내)
- mc-clip/ref-clip/빈 스파인 등 기존 `raise ValueError` 경로 → **작업은 계속**,
  해당 항목 통과 + 요약에 "분석 제외 N개 (멀티캠 등)" 안내
- 앱(Swift): 진단 로그에 "분석 대상 / 통과 항목" 카운트 표기
- 검증: mc-clip 픽스처가 에러 없이 결과물 생성하는지

### Phase 6 — 앱·문서 정합
- Swift: 안내 문구를 새 정책으로 ("메인 스토리라인 오디오 기준" 온보딩·README·
  COMPATIBILITY 매트릭스 §3으로 교체), 진단 로그 갱신
- 테스트 재정렬: 픽스처 기대값을 §3 계약 기준으로 조정
  (예: 커넥티드 오디오 자막 기대 제거)
- macOS/FCP 버전 매트릭스 재실행 (1.9~1.14)

### Phase 7 — v0.12.0 릴리스
- 전체 회귀(12종+Bible+버전 매트릭스+실파일+번들 런타임 E2E) → DMG → 릴리스
- 릴리스 노트: 새 동작 원칙("메인 스토리라인 오디오 기준")을 테스터에게 명확 안내
- TRACKING/메모리 갱신

## 6. 이번에 하지 않는 것 (명시적 제외)

- 커넥티드 오디오 분석 (원칙 2로 영구 제외가 아니라 v0.12 범위 밖)
- 2줄 자막 (기존 결정: 추후)
- 미디어 파일(mp4/wav) 직접 드롭 → 새 프로젝트 생성: **v0.12.1 후보**로 보류
  ("파일을 던져주면 분석"이라는 본질에 부합 — `fcpxml.py` 생성 경로 재활용 가능.
  원하시면 v0.12.0에 포함 가능)
- 세부 옵션 재오픈 (집중 테스트 유지, `betaFocusTest` 그대로)

## 8. 검증 기준 (완료 정의)

1. 픽스처 12종 + Bible 7종 + 버전 매트릭스(1.9~1.14) + 실제 커피국수 + 번들 런타임
   E2E — **전부 통과**
2. mc-clip/ref-clip 포함 프로젝트가 **에러 없이** 결과물 생성 (통과 + 안내)
3. `retranscribe.py` 코어 라인 수 **눈에 띄게 감소** (복잡성 감축의 정량 증거)
4. 새 지원 매트릭스(§3)가 README·COMPATIBILITY·온보딩에 일관 반영
